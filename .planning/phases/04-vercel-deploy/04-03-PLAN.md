<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User browser → cordyscrafts.vercel.app | All traffic over HTTPS (Vercel provides TLS by default) |
| cordyscrafts.vercel.app → Supabase | API calls from browser to Supabase using anon key (public, RLS-controlled) |
| WhatsApp link click → wa.me | Redirects user's phone to WhatsApp with pre-filled message; no data transmitted server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Information Disclosure | wa.me URL in browser history | accept | The wa.me link contains order details in the query string. This is visible in the browser URL bar and history on the user's device. Acceptable for v1 — no PII beyond name and phone number (which the user typed in themselves). |
| T-04-08 | Tampering | localStorage cart data | accept | Cart data in localStorage is client-side only and can be modified by the user. This is acceptable — there is no server-side cart; the cart contents are re-submitted via WhatsApp message which Cordeelia reviews manually. No financial risk from cart tampering. |
| T-04-09 | Denial of Service | Supabase anon INSERT | accept | RLS policies (Phase 2) allow anon INSERT to orders/bookings. A malicious actor could spam fake orders. Risk is low for v1 (small craft brand, no automated fulfillment). Mitigation deferred to v2 (rate limiting, CAPTCHA). |
| T-04-10 | Spoofing | WhatsApp message sender | accept | The pre-filled message is sent from the customer's own WhatsApp account. Spoofing would require the attacker to have access to the customer's phone. Not a platform-level risk. |
</threat_model>
