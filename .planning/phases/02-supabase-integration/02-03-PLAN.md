```js
const submit = async (e) => {
  e.preventDefault();
  if (!form.name || !form.phone) return;
  setSubmitting(true);
  setSubmitError(null);

  // Strip img and cat — orders.items JSONB schema requires only { sku, name, qty, price }
  const itemsForInsert = c.items.map(i => ({
    sku: i.sku, name: i.name, qty: i.qty, price: i.price,
  }));
  const totalForInsert = c.total();
  const itemsSnapshot = [...c.items]; // capture before c.clear() for the confirmation panel

  const { error } = await supabase.from('orders').insert({
    customer_name: form.name,
    customer_whatsapp: form.phone,
    address: form.addr || '',
    items: itemsForInsert,
    total_amount: totalForInsert,
    payment_status: 'pending',
  });

  if (error) {
    setSubmitError('Something went wrong — please try again.');
    setSubmitting(false);
    return; // wa.me link NEVER opens on insert failure (Phase 2 success criterion #2)
  }

  // INSERT succeeded → build WhatsApp message and open it
  const msg = buildOrderMessage(form, itemsSnapshot, totalForInsert);
  const waUrl = buildWaUrl(msg);
  window.open(waUrl, '_blank');

  setConfirmation({ ...form, total: totalForInsert, items: itemsSnapshot });
  setStage('done');
  c.clear();
  setSubmitting(false);
};
```

Implementation notes:
1. `address: form.addr || ''` — orders.address is NOT NULL per schema, so the column needs a string. Empty string is acceptable (the WhatsApp message helper renders 'Not provided' as the user-facing fallback).
2. The `itemsSnapshot` is captured BEFORE the INSERT and reused for both the buildOrderMessage call and setConfirmation. This is necessary because c.clear() runs at the end of the success branch — without the snapshot the confirmation panel would render an empty cart.
3. `supabase.from('orders').insert(...)` — does NOT chain `.select()`. Per Pitfall 1 in 02-RESEARCH.md, .select() is unnecessary (we don't need the inserted row id) and creates additional RLS surface (anon has no SELECT policy on orders). The destructured result is `{ data: null, error }` on success — check error only.
4. `window.open(waUrl, '_blank')` — Anti-Pattern in 02-RESEARCH.md flags `window.location.href` as wrong for wa.me on mobile (it would navigate the SPA away from localhost:5173 / production). `_blank` opens the WhatsApp app while keeping the SPA loaded.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user form input → orders/bookings INSERT | Customer name, phone, address pass directly from React state into Supabase JSONB and text columns; no server-side validation beyond HTML5 required + tel pattern |
| Supabase JS client → orders/bookings tables | RLS policy allows INSERT only; client cannot SELECT/UPDATE/DELETE — verified in Plan 01 checkpoint |
| Browser → user's WhatsApp app | wa.me URL is opened via window.open; the message body is built client-side and not validated by Supabase or WhatsApp |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Information Disclosure | submit handlers handle PII (name, phone, address) | mitigate | No console.log statements in either handler — verified by acceptance criteria. Error messages shown to the user are the locked string only ("Something went wrong — please try again.") with NO leaked Supabase error details. Per CLAUDE.md and 02-RESEARCH.md V7 Error Handling. |
| T-02-03-02 | Tampering | Anon client could attempt to insert orders.payment_status='paid' to bypass payment | mitigate | The INSERT payload always sends `payment_status: 'pending'` literally. The anon role has no UPDATE policy on orders, so once inserted, the field cannot be changed by the anon client. (Server-side enforcement of "anon may only insert payment_status='pending'" via RLS WITH CHECK is a future hardening — Phase 2 trusts the client to send 'pending' because there is no payment flow yet.) |
| T-02-03-03 | Tampering | Anon client could insert bookings with class_id pointing at a non-existent class | mitigate | bookings.class_id is a NOT NULL FK to classes(id). Postgres rejects FK violations at the constraint level — the INSERT returns an error which the guard catches, blocking wa.me. The Phase 2 supabase.test.js test "returns ok:false on FK violation" encodes this. |
| T-02-03-04 | Repudiation | Anon could spam orders/bookings | accept | Same disposition as T-02-01-05 — no rate limiting in Phase 2; Supabase Free Tier provides per-IP throttling that is sufficient for a tiny solo business launch. |
| T-02-03-05 | Tampering | wa.me URL constructed client-side could be modified before window.open | accept | An attacker who has DevTools access to their own browser session can already do anything they want; spoofing the wa.me message only affects what THEIR WhatsApp app shows when they tap Send. Cordeelia treats every WhatsApp message as input to be reviewed manually — there is no auto-acceptance based on URL contents. |
| T-02-03-06 | Denial of Service | Slow/hung Supabase request leaves submit button forever disabled | mitigate | The submit button has disabled={submitting}. If Supabase takes long enough that the user waits, the spinner-equivalent ("Sending...") informs them. There is no timeout/abort logic in Phase 2 — accepted because Supabase JS v2 has its own default timeouts (~30s) that will eventually surface as `error` and re-enable the button. |
| T-02-03-07 | Information Disclosure | Browser DevTools can inspect the supabase JS client and read VITE_SUPABASE_ANON_KEY | accept | The anon key is public by Supabase design (it's the key embedded in every browser bundle). RLS protects the data, not the key. T-02-02-04 already accepted. |
